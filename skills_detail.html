MWS Skills 广场
全家桶运营体系

从 0 到 1 建立 MWS Skills 生态的运营机制——制定准入规范、安全扫描标准、场景化运营策略，以及覆盖「主推 → 维持 → 迁移 → 淘汰」全周期的生命周期治理方案，推动 Skills 从野蛮生长走向有序运营。

Background
生态快速扩张后的治理困境
MWS Skills 广场是美团商家端的 AI 能力聚合平台，随着接入 Skill 数量快速增长，平台陷入了「有量无质」的困境：缺乏统一的质量标准，低质量/低活跃 Skill 大量堆积，安全审查机制缺失，且没有任何退出机制，废弃资产长期占用平台资源。
❌ 无统一准入标准 ❌ 安全审查机制缺失 ❌ 低活跃 Skill 大量堆积 ❌ 缺乏退出 / 下线机制 ❌ 商家对 Skills 认知度低 ✅ 建立准入规范体系 ✅ 安全扫描纳入上线流程 ✅ 四分层生命周期治理 ✅ 软下线机制保障有序退场
Strategy
三条主线并行推进
① 建规范

制定准入标准，明确质量底线

从基础信息、能力质量、安全合规、体验质量四个维度出发，建立必选 / 推荐 / 可选三级准入指标体系，形成可量化的上架评审标准。

② 建治理

生命周期分层，存量动态优化

建立「主推 / 维持 / 迁移 / 淘汰」四分层模型，配套软下线机制，让 Skill 资产能够有序流转，而非无限堆积。

③ 建认知

场景化运营 + 品牌扩张

通过「MWS 官方 Skills 全家桶」品牌化包装与多渠道触达策略，提升商家端对 Skills 的认知度与使用意愿。

Skills 广场的本质不是工具的简单陈列，而是场景化分发 + 质量保障 + 生态激励的有机体。运营的核心是让每个上架的 Skill 都有存在的理由，让每个离场的 Skill 都有体面的退出方式。
Admission Standards
准入规范体系
制定 MWS Skills 上架准入标准，分必选（must）、推荐（recommended）、可选（optional）三级，覆盖基础信息、能力质量、安全合规、体验质量四个维度。
维度检查项级别说明
基础信息 名称、描述、使用场景 必选 名称简洁，描述清晰说明能做什么
基础信息 头图 / 封面 必选 符合设计规范要求
能力质量 核心功能可用性验证 必选 上架前须通过基本冒烟测试
能力质量 响应时间 必选 满足平台响应时间基线要求
安全合规 安全扫描通过 必选 mws-skill-kit 扫描无高危项
安全合规 数据权限声明 必选 明确申报所访问的数据类型与范围
体验质量 错误处理与降级策略 推荐 提供友好的错误提示，支持兜底回复
体验质量 多轮对话支持 推荐 能识别上下文意图，避免频繁重复询问
生态共建 接入数据统计 推荐 上报使用指标，便于平台健康度监控
生态共建 开发者文档 可选 提供调试说明，方便审核与集成测试
Security
安全扫描机制（mws-skill-kit）
围绕 AI Skill 特有的安全风险，建立自动化扫描能力，纳入上架 CI 流程，新 Skill 上线前须通过检测。
🛡️

提示词注入检测

识别 System Prompt 中的恶意指令覆盖、角色劫持等攻击模式，防止 Skill 被外部输入绕过安全约束。

🔐

敏感数据泄露扫描

检测 Skill 输出是否包含用户隐私、商家经营数据等敏感字段，防止数据越界流出。

⚖️

内容合规性校验

对 Skill 功能描述与输出内容进行合规审查，过滤违禁信息、虚假宣传等违规表述。

🔗

外部依赖风险评估

扫描 Skill 调用的外部 API / 工具链，识别不可信来源与接口变更风险。

Lifecycle Management
四分层生命周期模型
针对存量 Skill 资产膨胀问题，建立「主推 / 维持 / 迁移 / 淘汰」四分层模型，定期对 Skill 进行健康度评估并动态调整分层状态。
主推（Active）
重点资源倾斜

高质量、高活跃、用户反馈好的 Skill，享有搜索加权、场景内置推荐、首页精选等平台资源倾斜。

搜索加权 场景内置 精选推荐
维持（Hold）
自然流量，无额外投入

功能基本可用，活跃度或质量评分未达主推门槛。保持正常上架状态，不主动推荐，依赖用户搜索发现。

正常展示 支持搜索 定期复审
迁移（Migrate）
能力合并 / 替代

核心能力已被更优 Skill 覆盖，或与其他 Skill 高度重叠。平台联系开发者完成能力迁移，展示替代推荐引导用户转移。

开发者通知 替代引导 过渡期保护
淘汰（Retire）
软下线 → 正式下架

长期低活跃、高错误率或已停止维护，经评估无改善可能。按软下线流程完成用户过渡后正式下架。

停止推荐 软下线通知 保护期 正式下架
Soft Deprecation
软下线机制
进入「淘汰」状态的 Skill 不直接强制下架，而是经历完整的软下线流程，保障存量用户使用连续性。
Step 1

判定触发

系统检测到 Skill 满足淘汰条件（持续低活跃、高错误率或长期无维护），由运营人工确认后启动流程。

Step 2

开发者通知（T+0）

通过大象消息通知 Skill 开发者，说明下架原因，提供改善窗口期，开发者可申诉或提交改进计划。

Step 3

软下线状态

Skill 进入「即将下架」状态：停止对新用户推荐，存量用户仍可正常使用；详情页展示下架倒计时提示，引导迁移至替代方案。

Step 4

正式下架

保护期结束后，Skill 从广场下架，历史使用数据归档留存。开发者可申请恢复，但须重新通过准入审核。
